# Agentic Architecture Migration Strategy
## Complete Migration Guide for Genesis Project

This document provides a comprehensive 7-phase strategy to migrate the Genesis Project from a form-based Shopify store generator into a production-ready agentic SaaS platform powered by conversational AI.

## Migration Overview

**Current State:** Form-based store configuration with Supabase backend
**Target State:** Agent-driven store configuration with approval workflows using OpenAI's latest capabilities
**Architecture:** Preserve all existing functionality while adding conversational AI layer

## 🚨 CRITICAL: Sequential Implementation Required

**MUST follow phases in exact order.** Each phase has explicit boundaries to prevent overengineering and feature creep. Do not implement features from future phases, even if they seem related or "easy to add now."

**Why sequential?** Each phase builds specific foundations that later phases depend on. Jumping ahead will result in:
- Overengineered early phases that are hard to maintain
- Missing critical infrastructure for later phases  
- Integration problems between components
- Poor performance and user experience

## Phase 1: Database Foundation & Tool Infrastructure

This phase establishes the database foundation and tool layer for AI agent operations.

### Context
- Current system: Form-based store configuration with Supabase backend
- Target: Agent-driven store configuration with approval workflows
- Database: Already has stores, store_data, uploads, collections, etc (refer to SUPABASE_SETUP.md)

### Key Requirements
1. **Add single comprehensive agent logs table:**
   - `agent_logs` (comprehensive audit log including conversations, tool calls, execution results, and before/after state)
   - **Important**: Each log entry represents one conversation turn and contains an ARRAY of tool calls in execution order
   - Multiple tool calls from a single user message are grouped in one log entry with `tool_call_id` (sequence-based IDs: 1, 2, 3...)
   - Each tool call has individual `operation_status` field:
     - **Modification tools**: pending → approved (when executed) OR rejected
     - **Read-only tools**: always "read" (static, no approval needed)
   - **No overall operation_status** - users can approve/reject individual tool calls
   - Keep pending changes in local browser state/session storage (no database persistence needed)

2. **Create comprehensive tool layer endpoints:**
   - `/api/tools/store-data` - CRUD operations on store configuration
   - `/api/tools/collections` - Collection management tools
   - `/api/tools/uploads` - File upload and management tools
   - `/api/tools/layout` - Store layout section management
   
3. **Add read-only tools for agent context gathering:**
   - `/api/tools/read-store-layout` - Get current store layout
   - `/api/tools/read-store-collections` - Get collections and mappings
   - `/api/tools/read-store-uploads` - Get uploaded files info
   - `/api/tools/read-store-visuals` - Get brand colors/fonts
   - `/api/tools/read-store-policies` - Get store policies

4. **Each tool endpoint must:**
   - Accept structured parameters with validation
   - Generate before/after diffs for changes (only changed fields, not full records)
   - Log all operations for audit trail
   - Return standardized response format with operation metadata

5. **Before/after state logging guidelines:**
   - **DO include**: Only fields that actually changed, table/record IDs
   - **DO NOT include**: Full database snapshots, unchanged fields, sensitive data (API tokens), large binary data
   - **Modification tool format**: `{"tool_call_id": 1, "tool_name": "updateStoreData", "operation_status": "pending", "before": {...}, "after": {...}}`
   - **Read-only tool format**: `{"tool_call_id": 2, "tool_name": "readStoreLayout", "operation_status": "read", "result": {...}}`
   - **Multi-tool log entry example**: `{"tool_calls": [{"tool_call_id": 1, "tool_name": "readStoreLayout", "operation_status": "read", "result": {...}}, {"tool_call_id": 2, "tool_name": "updateStoreData", "operation_status": "approved", "before": {...}, "after": {...}}]}`

### Implementation Tasks
1. Update `SUPABASE_SETUP.md` with single `agent_logs` table schema (include example JSON structure for multi-tool arrays with individual operation_status)
2. Create base tool infrastructure in `/lib/tools/` with efficient diff generation utilities
3. **Set up MCP (Model Context Protocol) server** - `/lib/mcp/` for dynamic tool calling during agent reasoning
   - **Building MCP Tutorial**: https://modelcontextprotocol.io/tutorials/building-mcp-with-llms
   - **MCP Full Documentation**: https://modelcontextprotocol.io/llms-full.txt (comprehensive reference)
   - **MCP TypeScript SDK**: https://github.com/modelcontextprotocol/typescript-sdk
   - **MCP Server Quickstart**: https://modelcontextprotocol.io/quickstart/server
4. Implement tool endpoints with proper error handling and audit logging
5. Add TypeScript interfaces for all tool operations, multi-tool sequences, and pending changes state
6. Create tool response formatting utilities and local state management for pending changes
7. Implement diff utilities that capture only changed fields (not full record snapshots)
8. Design logging utilities to group multiple tool calls from single conversation turn into ordered arrays with `tool_call_id`
9. **Build individual tool call approval/rejection functionality** - `/api/agent/approve-individual` and `/api/agent/reject-individual` endpoints using composite reference `{log_entry_id, tool_call_id}`
10. **Configure MCP tool definitions** for read-only context gathering (read-store-layout, read-store-visuals, etc.)

### Success Criteria
- All existing functionality preserved through new tool endpoints
- Complete audit trail for every operation with clear status tracking:
  - **Modification tools**: pending/approved/rejected status with before/after diffs
  - **Read-only tools**: "read" status with result data
- Individual tool call approve/reject functionality for modification tools only
- **MCP server operational** with tool definitions for read-only context gathering
- Foundation ready for OpenAI + MCP integration

### Design Rationale
- **Single audit table**: Simpler to implement and maintain initially; can be split into specialized tables later if needed
- **Local pending changes**: Faster UX (no database round-trips), simpler cleanup (auto-cleared on page refresh), and reduced database complexity
- **Individual tool call status**: Users can approve some changes and reject others from the same conversation - no overall status makes sense
- **UI-driven references**: Each approve/reject button carries `{log_entry_id, tool_call_id}` - direct, efficient, no searching needed
- **MCP Integration**: Enables true mid-response tool calling for natural agent reasoning

### ⚠️ PHASE 1 BOUNDARIES - DO NOT IMPLEMENT:
- **No OpenAI integration** (even though tools should be "ready" for it)
- **No agent orchestration or chat functionality**
- **No UI changes** to existing forms or layouts
- **No pending changes UI** (only the local state management utilities)
- **No diff visualization components** (only diff generation utilities)
- **No conversation interfaces**

**Focus ONLY on:** Tool endpoints, audit logging, and basic infrastructure.

## Phase 2: OpenAI Integration & Agent Orchestration

Building on Phase 1's tool foundation, implement the core AI agent using OpenAI's latest capabilities including the Responses API and function calling.

### Context
- Tool layer is now established with audit logging
- Need to integrate OpenAI's "gpt-5-mini-2025-08-07" model
- Must support natural language conversation and structured tool calling

### Key Requirements
1. **OpenAI + MCP Integration:**
   - Research and implement OpenAI Responses API (https://platform.openai.com/docs/api-reference/responses/create)
   - Research OpenAI Agents SDK (https://openai.github.io/openai-agents-js/)
   - **Reference**: Official OpenAI JavaScript/TypeScript SDK: https://github.com/openai/openai-node
   - **Integrate MCP (Model Context Protocol)** for mid-response tool calling during agent reasoning
   - **MCP Integration Resources**:
     - **Building MCP Guide**: https://modelcontextprotocol.io/tutorials/building-mcp-with-llms
     - **MCP Full Documentation**: https://modelcontextprotocol.io/llms-full.txt
   - Configure function calling with MCP tool definitions for dynamic context gathering

2. **Agent Core (`/lib/agent/`):**
   - `MCPEnabledOrchestrator` class for conversation management with mid-response tool calling
   - Tool calling interface that integrates OpenAI streaming with MCP server
   - Context management for multi-turn conversations with dynamic data reading
   - Local state management for pending changes (React state/session storage)
   - **MCP streaming handler** for true mid-response tool execution and continued reasoning

3. **Agent API endpoints:**
   - `/api/agent/chat` - Main conversation endpoint with streaming support (https://platform.openai.com/docs/guides/streaming-responses?api-mode=responses&lang=javascript)
   - `/api/agent/approve-individual` - Approve individual tool call using `{log_entry_id, tool_call_id}` reference
   - `/api/agent/reject-individual` - Reject individual tool call using `{log_entry_id, tool_call_id}` reference
   - `/api/agent/approve-all` - Approve ALL pending tool calls from a conversation turn
   - `/api/agent/reject-all` - Reject ALL pending tool calls from a conversation turn  
   - `/api/agent/history` - Conversation history (from agent_logs table)

### Implementation Details
1. **Agent personality and instructions:**
   - Expert Shopify store consultant
   - Proactively suggests improvements
   - Always explains changes before proposing them
   - Maintains context across conversation turns

2. **MCP + OpenAI Function calling structure:**
   - Map read-only tools to MCP tool definitions for mid-response context gathering
   - Map modification tools to OpenAI function definitions for structured changes
   - Include parameter validation and descriptions for all tools
   - Handle complex multi-step operations with MCP tool chaining
   - **Use OpenAI streaming responses** with MCP integration for mid-response reasoning

3. **MCP-Enhanced Conversation flow:**
   - Accept natural language input with minimal context
   - Agent reasons and calls MCP read tools as needed ("Let me check your current layout...")
   - **Mid-response tool calling**: Agent can gather context, continue reasoning, and propose changes
   - Execute modification tool calls in order and log them as an array in single conversation entry
   - Create structured diffs from modification tools only (read tools don't generate diffs)
   - Present changes for approval
   - Execute approved changes

### Success Criteria
- Agent can understand store configuration requests
- **MCP integration functional** - agent can call read tools mid-response and continue reasoning
- Proper function calling to tool endpoints (single and multi-tool sequences)
- OpenAI streaming responses with MCP tool calling implemented (complete responses logged after streaming)
- Conversation history persistence with grouped tool call logging (including MCP read calls)
- Change proposal and approval workflow functional
- Test via API endpoints or simple command-line interface showing mid-response tool calling

### ⚠️ PHASE 2 BOUNDARIES - DO NOT IMPLEMENT:
- **No chat UI components** (no React components for conversation)
- **No modifications to existing form pages**
- **No visual diff components** (only diff data generation)
- **No real-time UI streaming** (use OpenAI streaming for agent responses, but don't display streaming to users yet)
- **No rainbow outline effects or visual feedback**
- **No form integration or sync**

**Focus ONLY on:** Agent logic, OpenAI integration, and API endpoints.

## Phase 3: Chat Interface & User Experience

With the agent core functional, create the conversational UI that integrates alongside the existing store configuration form.

### Context
- Agent API is working with tool calling and approval workflows
- Existing store form UI must remain functional
- Need to add conversational interface without disrupting current UX

### Key Requirements
1. **MCP-Enhanced Chat Interface Components (`/components/chat/`):**

   - `MessageInput` - Natural language input
   - `MCPStreamingDisplay` - Real-time display of agent reasoning and MCP tool calls
   - `PendingChanges` - Visual diff preview for modification tools only (each carrying `{log_entry_id, tool_call_id}` reference)  
   - `ApprovalControls` - Individual and "Approve/Reject ALL" controls (modification tools only)
   - `ToolCallIndicator` - Visual indicators for ALL tool calls:
     - **Read operations**: "🔍 Reading store layout..." (informational only, no approval needed)
     - **Modification operations**: "✏️ Updating store data..." (shows what's being proposed)
   - `AgentTyping` - Loading states and typing indicators

2. **Layout Integration:**
   - Modify `/app/dashboard/clients/[clientId]/page.tsx` to include chat panel
   - Responsive design: chat panel alongside form on desktop, tabbed on mobile
   - Maintain existing form functionality completely

3. **MCP-Enhanced Real-time Features:**
   - **Build on Phase 2's OpenAI + MCP streaming** - display agent reasoning and mid-response tool calls to users in real-time (https://platform.openai.com/docs/guides/streaming-responses?api-mode=responses&lang=javascript)
   - Progressive display of agent reasoning: "Let me check your current layout..." → [calls read-store-layout] → "I can see you have..."
   - **Visual indicators for MCP tool calls** - show when agent is gathering context vs. proposing changes
   - Optimistic UI updates for smooth experience

### Implementation Details
1. **MCP-Enhanced Chat UI Features:**
   - Message threading with conversation context
   - **Real-time streaming display** of agent reasoning with tool call indicators for ALL operations
   - Rich diff visualization for proposed changes (modification tools only)
   - **Tool call visualization** - show ALL operations as they happen:
     - Read operations: "🔍 Checking current layout..."
     - Modification operations: "✏️ Updating store colors..."
   - Inline previews of store configuration changes
   - **Individual approve/reject controls per modification tool call** - each button carries `{log_entry_id, tool_call_id}` for direct `/api/agent/approve-individual` or `/api/agent/reject-individual` API calls
   - **Read-only tool call display** - show MCP read operations as informational only (no approval controls)
   - "Approve ALL" and "Reject ALL" buttons for batch operations on modification tools only
   - Quick actions for common operations

2. **Form Integration:**
   - Visual indicators when agent has modified specific fields
   - Rainbow outline effect for AI-modified fields (as specified)
   - Sync between agent changes and form state
   - Maintain auto-save functionality

3. **User Experience:**
   - Onboarding flow for new users
   - Example prompts and suggestions
   - Error handling and graceful fallbacks
   - Conversation persistence across sessions

4. **Tool Call UX Flow Examplle (similar to Cursor IDE):**
   - **Agent starts reasoning**: "Let me check your current store setup..."
   - **Read tool indicator appears**: "🔍 Reading store layout..."
   - **Read tool completes**: "I can see your store has a slideshow hero section..."
   - **Modification tool indicator appears**: "✏️ Updating store colors..."
   - **Modification results**: "I've proposed the following changes:" → shows diff with approval buttons
   - **All operations tracked and visible** throughout the conversation

### Success Criteria
- Chat panel appears alongside existing form with MCP streaming support
- Users can make store configuration requests in natural language
- **Real-time display of agent reasoning** with tool call indicators for ALL operations:
  - Read operations: "🔍 Checking current layout..." → "I can see you have..."
  - Modification operations: "✏️ Updating store colors..." → "I've proposed the following changes..."
- Clear distinction between tool types:
  - **Read operations**: Visible as informational, "read" status, no approval needed
  - **Modification operations**: Show diffs with pending status, require approval
- All existing form functionality preserved
- Visual indicators for AI-modified fields working
- Responsive design on all screen sizes

### ⚠️ PHASE 3 BOUNDARIES - DO NOT IMPLEMENT:
- **No advanced diff visualization** (basic diff preview only)
- **No rainbow outline effects** (save for Phase 6)
- **No form-agent deep integration** (chat panel is separate)
- **No advanced tool operations** (stick to basic store configuration)
- **No analytics or monitoring dashboards**

**Focus ONLY on:** MCP-enhanced chat UI with real-time tool call display, simple diff previews, and conversation interface.

## Phase 5: Structured Diff System & Change Management

Implement a sophisticated diff and approval system that provides clear visibility into all proposed changes with granular control.

### Context
- Agent can propose and execute changes via MCP-enhanced workflows
- Need production-ready change management system
- Must provide clear visibility and control over all modifications
- **Distinguish between MCP read operations** (context gathering, no approval needed) **and modification tools** (require approval)

### Key Requirements
1. **Advanced Diff System (`/lib/diff/`):**
   - Structured diff generation for all data types
   - Visual diff representation for complex objects
   - Field-level change tracking with metadata
   - Clear before/after state visualization

2. **Approval Workflow:**
   - Granular approval (approve/reject individual changes)
   - "Approve ALL" and "Reject ALL" batch operations
   - Change review with explanation and reasoning
   - Rollback capabilities for applied changes

3. **Change Visualization:**
   - Interactive diff viewer component
   - Side-by-side comparison for complex changes
   - Clear highlighting of changed vs. unchanged fields
   - User-friendly change summaries

### Implementation Details
1. **Diff Engine:**
   - Deep object comparison with type-aware diffing
   - JSON patch format for programmatic changes
   - Human-readable change descriptions

2. **UI Components:**
   - `DiffViewer` - Main change visualization (modification tools only)
   - `ModificationToolCallCard` - Individual modification tool call with `{log_entry_id, tool_call_id}` reference and approval controls (pending/approved/rejected status)
   - `ReadToolCallDisplay` - Display MCP read operations in chat (always "read" status, informational only, no approval needed)
   - `ModificationToolCallDisplay` - Display MCP modification operations in chat (shows tool call indicator, then results in pending changes)
   - `ApprovalControls` - Individual and "Approve/Reject ALL" interface for modification tools only
   - `ChangeHistory` - Historical change log with clear distinction:
     - **Read operations**: "read" status, show result data
     - **Modification operations**: pending/approved/rejected status, show diffs

3. **Workflow Management:**
   - Change queuing and dependency resolution
   - Atomic operations (all-or-nothing changes)
   - Change scheduling and delayed execution
   - Notification system for pending approvals

### Advanced Features
1. **Smart Grouping:**
   - Related changes grouped logically
   - Suggested approval flows

2. **Change Analytics:**
   - Track most common change types
   - Success/failure rates for different operations
   - User preference learning

### Success Criteria
- Clear visualization with distinct handling for tool types:
  - **Modification tools**: Show diffs with pending/approved/rejected status and approval controls
  - **Read-only tools**: Show results with static "read" status (informational only)
- Granular control over modification tool approval/rejection using explicit endpoints with `{log_entry_id, tool_call_id}` references
- Batch (Approve/Reject ALL) control for modification tools only
- Reliable rollback capabilities for approved modifications
- Complete change history and audit trail with clear tool type distinction
- Intuitive user experience - users understand which actions need approval vs. informational reads
- Production-ready error handling and validation

**Focus on building trust through transparency and control.**

## Implementation Strategy

**Execute these phases sequentially,** with each phase building on the previous one. Each prompt includes:

1. **Clear context** about what has been accomplished
2. **Specific requirements** for the current phase
3. **Implementation details** with file structure guidance
4. **Success criteria** for validation
5. **Production considerations** throughout

**Key Principles:**
- Preserve all existing functionality while adding agentic capabilities
- Production-grade implementation with proper error handling and monitoring
- Extensible architecture ready for multiple data domains (products, inventory, etc.)
- Complete audit trail and approval workflows as specified
- Modern OpenAI integration using "gpt-5-mini-2025-08-07" and latest APIs

**Expected Outcome:** A fully functional agentic SaaS platform where users can configure Shopify stores through natural language conversations, with every change requiring explicit approval and providing clear visual feedback for AI modifications.

The migration maintains backward compatibility while building the foundation for conversational AI-powered store management that can later extend to product data editing, inventory management, and other domains.

✅ **Phase Completion:** All success criteria met for current phase
