# Agentic Architecture Migration Strategy
## Complete Migration Guide for Genesis Project

This document provides a comprehensive 6-phase strategy to migrate the Genesis Project from a form-based Shopify store generator into a production-ready agentic SaaS platform powered by conversational AI.

## Migration Overview

**Current State:** Form-based store configuration with Supabase backend
**Target State:** Agent-driven store configuration with approval workflows using OpenAI's latest capabilities
**Architecture:** Preserve all existing functionality while adding conversational AI layer

## 🚨 CRITICAL: Sequential Implementation Required

**MUST follow phases in exact order.** Each phase has explicit boundaries to prevent overengineering and feature creep. Do not implement features from future phases, even if they seem related or "easy to add now."

**Why sequential?** Each phase builds specific foundations that later phases depend on. Jumping ahead will result in:
- Overengineered early phases that are hard to maintain
- Missing critical infrastructure for later phases  
- Integration problems between components
- Poor performance and user experience

## Phase 1A: Database & Audit Foundation

Establish the database foundation and audit logging system for agent operations.

### Context
- Current system: Form-based store configuration with Supabase backend
- Target: Agent-driven store configuration with approval workflows
- Database: Already has stores, store_data, uploads, collections, etc (refer to SUPABASE_SETUP.md)

### Key Requirements
1. **Add single comprehensive agent logs table:**
   - `agent_logs` (comprehensive audit log including conversations, tool calls, execution results, and before/after state)
   - **Important**: Each log entry represents one conversation turn and contains an ARRAY of tool calls in execution order
   - Multiple tool calls from a single user message are grouped in one log entry with `tool_call_id` (sequence-based IDs: 1, 2, 3...)
   - Each tool call has individual `operation_status` field:
     - **Modification tools**: pending → approved (when executed) OR rejected
     - **Read-only tools**: always "read" (static, no approval needed)
   - **No overall operation_status** - users can approve/reject individual tool calls
   - Keep pending changes in local browser state/session storage (no database persistence needed)

2. **Before/after state logging guidelines:**
   - **DO include**: Only fields that actually changed, table/record IDs
   - **DO NOT include**: Full database snapshots, unchanged fields, sensitive data (API tokens), large binary data
   - **Modification tool format**: `{"tool_call_id": 1, "tool_name": "updateStoreData", "operation_status": "pending", "before": {...}, "after": {...}}`
   - **Read-only tool format**: `{"tool_call_id": 2, "tool_name": "readStoreLayout", "operation_status": "read", "result": {...}}`
   - **Multi-tool log entry example**: `{"tool_calls": [{"tool_call_id": 1, "tool_name": "readStoreLayout", "operation_status": "read", "result": {...}}, {"tool_call_id": 2, "tool_name": "updateStoreData", "operation_status": "approved", "before": {...}, "after": {...}}]}`

### Implementation Tasks
1. Update `SUPABASE_SETUP.md` with single `agent_logs` table schema (include example JSON structure for multi-tool arrays with individual operation_status)
2. Create base infrastructure in `/lib/audit/` for logging utilities
3. Design logging utilities to group multiple tool calls from single conversation turn into ordered arrays with `tool_call_id`
4. Add TypeScript interfaces for audit logging, tool call structures, and pending changes state
5. Create local state management utilities for pending changes (React state/session storage)

### Success Criteria
- `agent_logs` table schema defined and documented
- Complete audit trail design with clear status tracking:
  - **Modification tools**: pending/approved/rejected status with before/after diffs
  - **Read-only tools**: "read" status with result data
- Local pending changes state management utilities created
- TypeScript interfaces defined for all logging structures
- Foundation ready for tool layer implementation

### Design Rationale
- **Single audit table**: Simpler to implement and maintain initially; can be split into specialized tables later if needed
- **Local pending changes**: Faster UX (no database round-trips), simpler cleanup (auto-cleared on page refresh), and reduced database complexity
- **Individual tool call status**: Users can approve some changes and reject others from the same conversation - no overall status makes sense
- **UI-driven references**: Each approve/reject button carries `{log_entry_id, tool_call_id}` - direct, efficient, no searching needed

### ⚠️ PHASE 1A BOUNDARIES - DO NOT IMPLEMENT:
- **No tool endpoints** (Phase 1B)
- **No MCP server setup** (Phase 2A)
- **No OpenAI integration** (Phase 2B)
- **No UI components or changes**
- **No agent orchestration or chat functionality**

**Focus ONLY on:** Database schema, audit logging design, and basic infrastructure.

## Phase 1B: Core Tool Infrastructure

Implement the comprehensive tool layer with 20 endpoints covering all form sections.

### Context
- Database foundation and audit logging are established
- Need to create the tool layer that agents will use to read and modify store data
- Must cover all 10 sections of the client form with read and modification tools

### Key Requirements
1. **Create comprehensive tool layer endpoints:**
   - **20 comprehensive tool endpoints** (10 read + 10 modification) covering all form sections
   - **Granular section-specific tools** instead of generic store-data operations
   - **File upload tools** for logo and CSV data handling
   - **Preserved existing features** like magic generate
   
2. **Add read-only tools for agent context gathering (10 tools covering all form sections):**
   - `/api/tools/read-store-info` - Get store name, description, contact email
   - `/api/tools/read-store-visuals` - Get brand colors, fonts
   - `/api/tools/read-store-logo` - Get logo upload info and current logo
   - `/api/tools/read-data-files` - Get CSV uploads info (products, customers, inventory)
   - `/api/tools/read-store-collections` - Get collections and mapping rules
   - `/api/tools/read-store-locations` - Get store location details
   - `/api/tools/read-shipping-logistics` - Get processing times and shipping options
   - `/api/tools/read-legal-info` - Get trading name, business details, VAT, registration numbers
   - `/api/tools/read-store-policies` - Get return policy, privacy policy, terms of service, shipping policy
   - `/api/tools/read-store-layout` - Get store page configurations

3. **Add modification tools for agent changes (10 tools covering all form sections):**
   - `/api/tools/update-store-info` - Update store name, description, contact email
   - `/api/tools/update-store-visuals` - Update brand colors, fonts
   - `/api/tools/upload-store-logo` - Handle logo upload/replacement
   - `/api/tools/upload-data-files` - Handle CSV uploads (products, customers, inventory)
   - `/api/tools/update-store-collections` - Manage product collections (preserve magic generate feature)
   - `/api/tools/update-store-locations` - Manage store location details
   - `/api/tools/update-shipping-logistics` - Update processing times and shipping options
   - `/api/tools/update-legal-info` - Update trading name, business details, VAT, registration numbers
   - `/api/tools/update-store-policies` - Update return policy, privacy policy, terms of service, shipping policy
   - `/api/tools/update-store-layout` - Update store page configurations

4. **Each tool endpoint must:**
   - Accept structured parameters with validation
   - Generate before/after diffs for changes (only changed fields, not full records)
   - Log all operations for audit trail
   - Return standardized response format with **UI-ready metadata**:
     - Human-readable tool descriptions and parameter names
     - Execution results formatted for display
     - Tool category (read vs modification) for proper UI handling
     - Parameter values with display formatting

### Implementation Tasks
1. Create base tool infrastructure in `/lib/tools/` with efficient diff generation utilities
2. **Implement comprehensive tool endpoints (20 total)** with proper error handling and audit logging:
   - **10 Read tools**: read-store-info, read-store-visuals, read-store-logo, read-data-files, read-store-collections, read-store-locations, read-shipping-logistics, read-legal-info, read-store-policies, read-store-layout
   - **10 Modification tools**: update-store-info, update-store-visuals, upload-store-logo, upload-data-files, update-store-collections, update-store-locations, update-shipping-logistics, update-legal-info, update-store-policies, update-store-layout
   - **Preserve magic generate feature** for collections section and policies section
   - **Section-specific validation** and error handling for each tool
3. Add TypeScript interfaces for all tool operations and responses
4. Create tool response formatting utilities
5. Implement diff utilities that capture only changed fields (not full record snapshots)
6. **Design tool interfaces with MCP compatibility in mind:**
   - Standardized parameter schemas (JSON Schema format)
   - Consistent return value structures
   - Standardized error response formats
   - Tool metadata (descriptions, parameter types) that can be easily mapped to MCP tool definitions
7. **Build individual tool call approval/rejection functionality** - `/api/agent/approve-individual` and `/api/agent/reject-individual` endpoints using composite reference `{log_entry_id, tool_call_id}`:
   - **Return updated log entry data** after status changes for UI refresh
   - Include tool call metadata for proper UI display
   - Provide batch status update information for "Approve/Reject ALL" operations

### Success Criteria
- **All 10 form sections covered** by dedicated read and modification tools (20 endpoints total)
- All existing functionality preserved (including magic generate)
- Individual tool call approve/reject functionality for modification tools only
- Comprehensive error handling and validation for all tools
- Efficient diff generation utilities working
- Foundation ready for MCP integration

### ⚠️ PHASE 1B BOUNDARIES - DO NOT IMPLEMENT:
- **No MCP server setup** (Phase 2A)
- **No OpenAI integration** (Phase 2B)
- **No UI components or changes**
- **No agent orchestration or chat functionality**
- **No diff visualization components** (only diff generation utilities)

**Focus ONLY on:** Tool endpoints, validation, and approval/rejection APIs.

## Phase 2A: MCP Server Setup

Establish the Model Context Protocol server for dynamic tool calling during agent reasoning.

### Context
- Tool infrastructure is established with 20 endpoints
- Need to set up MCP server for mid-response tool calling capabilities
- This foundation will enable natural agent reasoning with dynamic context gathering

### Key Requirements
1. **Set up MCP (Model Context Protocol) server:**
   - `/lib/mcp/` for dynamic tool calling during agent reasoning
   - **Building MCP Tutorial**: https://modelcontextprotocol.io/tutorials/building-mcp-with-llms
   - **MCP Full Documentation**: https://modelcontextprotocol.io/llms-full.txt (comprehensive reference)
   - **MCP TypeScript SDK**: https://github.com/modelcontextprotocol/typescript-sdk
   - **MCP Server Quickstart**: https://modelcontextprotocol.io/quickstart/server

2. **Configure MCP tool definitions for all 20 tools:**
   - **Read tools for MCP**: 10 tools for dynamic context gathering during agent reasoning
   - **Modification tools for OpenAI**: 10 tools for structured changes requiring approval
   - Proper parameter validation and descriptions for all tools
   - Tool categorization (read vs. modification) for proper handling

### Implementation Tasks
1. Install and configure MCP TypeScript SDK
2. Create MCP server configuration with tool definitions
3. **Create MCP adapter layer** that wraps existing Phase 1B REST endpoints:
   - MCP tools call internal REST APIs (no endpoint changes needed)
   - Transform REST API responses to MCP-compatible formats
   - Map existing parameter schemas to MCP tool definitions
4. Map all 20 tools to appropriate MCP/OpenAI interfaces:
   - **10 Read tools**: Available through MCP for mid-response context gathering
   - **10 Modification tools**: Available through OpenAI function calling for approval workflow
5. Set up MCP server connection and communication protocols
6. Create tool validation and error handling for MCP integration
7. Test MCP server functionality with basic tool calling

### Success Criteria
- **MCP server operational** with tool definitions for all 20 tools
- Read tools properly configured for dynamic context gathering
- Modification tools properly configured for approval workflow
- MCP server can handle tool calls and return structured responses
- Foundation ready for OpenAI integration
- Complete documentation of MCP setup and tool mappings

### ⚠️ PHASE 2A BOUNDARIES - DO NOT IMPLEMENT:
- **No OpenAI integration** (Phase 2B)
- **No agent orchestration logic** (Phase 2B)
- **No chat functionality** (Phase 3A)
- **No UI components**

**Focus ONLY on:** MCP server setup, tool definitions, and basic connectivity.

## Phase 2B: OpenAI Integration & Agent Core

Integrate OpenAI's latest capabilities with the MCP server to create the core AI agent.

### Context
- MCP server is operational with tool definitions
- Need to integrate OpenAI's "gpt-5-mini-2025-08-07" model
- Must support natural language conversation and structured tool calling

### Key Requirements
1. **OpenAI + MCP Integration:**
   - Research and implement OpenAI Responses API (https://platform.openai.com/docs/api-reference/responses/create)
   - Research OpenAI Agents SDK (https://openai.github.io/openai-agents-js/)
   - **Reference**: Official OpenAI JavaScript/TypeScript SDK: https://github.com/openai/openai-node
   - **Integrate MCP server** for mid-response tool calling during agent reasoning
   - Configure function calling with MCP tool definitions for dynamic context gathering

2. **Agent Core (`/lib/agent/`):**
   - `MCPOrchestrator` class for conversation management with mid-response tool calling
   - Tool calling interface that integrates OpenAI streaming with MCP server
   - Context management for multi-turn conversations with dynamic data reading
   - **MCP streaming handler** for true mid-response tool execution and continued reasoning

3. **Agent API endpoints:**
   - `/api/agent/chat` - Main conversation endpoint with streaming support (https://platform.openai.com/docs/guides/streaming-responses?api-mode=responses&lang=javascript)
   - `/api/agent/approve-all` - Approve ALL pending tool calls from a conversation turn
   - `/api/agent/reject-all` - Reject ALL pending tool calls from a conversation turn  
   - `/api/agent/history` - Conversation history (from agent_logs table)

### Implementation Details
1. **Agent personality and instructions:**
   - Expert Shopify store consultant
   - Proactively suggests improvements
   - Always explains changes before proposing them
   - Maintains context across conversation turns

2. **MCP + OpenAI Function calling structure:**
   - Integrate MCP server for read tool dynamic context gathering
   - Map modification tools to OpenAI function definitions for structured changes
   - Handle complex multi-step operations with MCP tool chaining
   - **Use OpenAI streaming responses** with MCP integration for mid-response reasoning
   - **Structure streaming events for UI parsing:**
     - Tool call start events with metadata (tool name, parameters)
     - Tool call completion events with results  
     - Clear separation between read and modification tool events
     - Progress indicators for multi-tool operations

3. **MCP-Enhanced Conversation flow:**
   - Accept natural language input with minimal context
   - Agent reasons and calls MCP read tools as needed ("Let me check your current layout...")
   - **Mid-response tool calling**: Agent can gather context, continue reasoning, and propose changes
   - Execute modification tool calls in order and log them as an array in single conversation entry
   - Create structured diffs from modification tools only (read tools don't generate diffs)
   - Present changes for approval
   - Execute approved changes

### Success Criteria
- Agent can understand store configuration requests
- **MCP integration functional** - agent can call read tools mid-response and continue reasoning
- Proper function calling to tool endpoints (single and multi-tool sequences)
- OpenAI streaming responses with MCP tool calling implemented (complete responses logged after streaming)
- **Streaming responses structured for UI consumption** - tool call events properly formatted for real-time display
- Conversation history persistence with grouped tool call logging (including MCP read calls)
- **Conversation API endpoints return UI-ready data** - log entries with complete tool call metadata for display
- Change proposal and approval workflow functional
- Test via API endpoints or simple command-line interface showing mid-response tool calling

### ⚠️ PHASE 2B BOUNDARIES - DO NOT IMPLEMENT:
- **No chat UI components** (Phase 3A)
- **No modifications to existing form pages** (Phase 3B)
- **No visual diff components** (Phase 3A)
- **No real-time UI streaming** (Phase 3B)
- **No rainbow outline effects or visual feedback** (Phase 3B)

**Focus ONLY on:** Agent logic, OpenAI + MCP integration, and API endpoints.

## Phase 3A: Basic Chat Interface

Create the core conversational UI components without real-time streaming features.

- Agent API is working with tool calling and approval workflows

   - `ApprovalControls` - Individual and "Approve/Reject ALL" controls (modification tools only)
   - `ToolCallIndicator` - **Streaming-compatible** visual indicators for tool calls:
     - **Read operations**: Support both states - "🔍 Checking layout..." and "🔍 Checked layout..."
     - **Modification operations**: Support both states - "✏️ Updating data..." and "✏️ Updated data..."
     - **Built with streaming state transitions in mind**
   - `AgentTyping` - Loading states and typing indicators (extensible for streaming)

2. **Layout Integration:**
   - Modify `/app/dashboard/clients/[clientId]/page.tsx` to include chat panel
   - Responsive design: chat panel alongside form on desktop, tabbed on mobile
   - Maintain existing form functionality completely

### Implementation Details
1. **Streaming-Ready Chat UI Foundation:**
   - Message threading with conversation context (supports progressive updates)
   - Basic diff visualization for proposed changes (modification tools only)
   - **Tool call visualization architecture** - components designed to handle state transitions:
     - **Phase 3A Implementation**: Use completed state only ("🔍 Checked current layout...")
     - **Phase 3A Architecture**: Built to support streaming states for Phase 3B enhancement
     - **State management**: Foundation supports both static display (3A) and streaming transitions (3B)
   - Inline previews of store configuration changes
   - **Individual approve/reject controls per modification tool call** - each button carries `{log_entry_id, tool_call_id}` for direct `/api/agent/approve-individual` or `/api/agent/reject-individual` API calls
   - **Read-only tool call display** - show MCP read operations as informational only (no approval controls)
   - "Approve ALL" and "Reject ALL" buttons for batch operations on modification tools only
   - Quick actions for common operations

2. **User Experience:**
   - Onboarding flow for new users
   - Example prompts and suggestions
   - Error handling and graceful fallbacks
   - Conversation persistence across sessions

### Success Criteria
- Chat panel appears alongside existing form
- Users can make store configuration requests in natural language
- **Clear display of completed tool operations** with tool call indicators for ALL operations:
  - Read operations: "🔍 Checked current layout..." → "I can see you have..."
  - Modification operations: "✏️ Updated store colors..." → "I've proposed the following changes..."
- Clear distinction between tool types:
  - **Read operations**: Visible as informational, "read" status, no approval needed
  - **Modification operations**: Show diffs with pending status, require approval
- All existing form functionality preserved
- Responsive design on all screen sizes
- Individual and batch approval/rejection working

## Implementation Strategy

**Execute these 6 phases sequentially,** with each phase building on the previous one. Each phase includes:

1. **Clear context** about what has been accomplished
2. **Specific requirements** for the current phase
3. **Implementation details** with file structure guidance
4. **Success criteria** for validation
5. **Clear boundaries** to prevent overengineering

**Phase Overview:**
- **Phase 1A**: Database & Audit Foundation (audit table, logging design)
- **Phase 1B**: Core Tool Infrastructure (20 tool endpoints, approval APIs)
- **Phase 2A**: MCP Server Setup (tool definitions, MCP configuration)
- **Phase 2B**: OpenAI Integration & Agent Core (agent logic, streaming)
- **Phase 3A**: Basic Chat Interface (core UI components, basic functionality)
- **Phase 3B**: Advanced Features & Integration (real-time streaming, form integration, rainbow outlines)

**Key Principles:**
- Preserve all existing functionality while adding agentic capabilities
- Production-grade implementation with proper error handling
- Extensible architecture ready for future enhancements
- Complete audit trail and approval workflows as specified
- Modern OpenAI integration using "gpt-5-mini-2025-08-07" and latest APIs
- **MCP-first approach** for true mid-response tool calling and reasoning

**Expected Outcome:** A fully functional agentic SaaS platform where users can configure Shopify stores through natural language conversations, with every change requiring explicit approval and providing clear visual feedback for AI modifications.

The migration maintains backward compatibility while building the foundation for conversational AI-powered store management that can later extend to product data editing, inventory management, and other domains.
